## Compatibility with Popular Skills

### With superpowers (obra/superpowers)

superpowers enforces a brainstorm вҶ’ spec вҶ’ plan вҶ’ build вҶ’ review workflow. init-project fits as the **spec generation step**:

```
superpowers /brainstorm вҶ’ /init-project (generates specs) вҶ’ superpowers /plan вҶ’ /build
```

The specs generated by init-project become the input that superpowers' planning phase consumes.

### With gstack (garrytan/gstack)

gstack organizes development as a virtual team (CEO вҶ’ Designer вҶ’ Eng вҶ’ QA вҶ’ Ship). init-project provides the **documentation foundation** that gstack's personas reference:

```
/init-project вҶ’ gstack /autoplan (reads CLAUDE.md + requirements.md) вҶ’ /ship
```

### With brainstorming skills

Run init-project **before** brainstorming to give the creative exploration a concrete project context. Without specs, brainstorming produces vague ideas. With specs, it produces actionable proposals.

### Standalone usage

init-project works perfectly on its own. The integrations above are optional enhancements, not requirements.

## When to Run Each Mode

| Situation | Mode | What happens |
|-----------|------|-------------|
| New project, no docs | `init` | Full 5-phase process |
| Existing project, no docs | `init` | Reads code first, then asks focused questions |
| Added new features/modules | `update` | Appends to requirements + progress, updates architecture |
| Changed tech stack | `update` | Updates CLAUDE.md, architecture, decisions |
| Before a release | `audit` | Cross-checks all documents for consistency |
| Onboarding new team member | `audit` | Validates docs are current and complete |
| After major refactor | `update` | Syncs structure, modules, and file references |
