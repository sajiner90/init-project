### D001: 语音通道选型
**日期：** 2026-03-25
**背景：** 需要全双工语音对话（用户说话时 AI 能听，AI 说话时用户能打断）
**考虑的方案：**
1. **WebSocket + 自建 ASR/TTS** — 灵活 / 工程量大，需自己处理 VAD、回声消除
2. **OpenAI Realtime API** — 一站式 / 国内延迟高，绑定 OpenAI 模型
3. **火山引擎 RTC** — 国内基础设施成熟，原生 ASR+TTS+VAD+打断 / SDK 文档一般
**结论：** 选择火山引擎 RTC。国内 RTT < 200ms，LLM webhook 模式可对接任意大模型
**风险：** SDK 文档质量一般，排查成本高。缓解：先跑通 demo 记录踩坑经验

### D002: LLM 分工
**日期：** 2026-03-25
**背景：** 问诊和练习对 LLM 的能力需求不同
**考虑的方案：**
1. **统一用 Claude** — 简单 / 问诊对话不需要 Agent 能力
2. **统一用 GPT** — 中文好 / 练习需要工具调用，GPT function calling 不如 Agent SDK
3. **分工：问诊 GPT + 练习 Claude** — 各取所长 / 维护两套 LLM 接口
**结论：** 分工方案。Claude Agent SDK 的工具调用适合练习引导，GPT 中文对话适合问诊

### D003: 前端框架
**日期：** 2026-03-26
**背景：** 需要 SSR + API Routes + 良好的 TypeScript 支持
**考虑的方案：**
1. **Vite + React** — 轻量 / 需要单独搭 API 服务
2. **Next.js 14 App Router** — SSR + API 一体化 / 学习曲线
3. **Remix** — 数据加载模式好 / 生态不如 Next
**结论：** Next.js 14。API Routes 避免额外后端服务，App Router 的 Server Components 减少客户端 JS

### D004: 数据库
**日期：** 2026-03-26
**背景：** 需要持久化用户数据、练习历史、体式库
**考虑的方案：**
1. **localStorage** — 零成本 / 不可跨设备，数据易丢失
2. **SQLite** — 嵌入式 / 不适合未来多用户
3. **PostgreSQL + Prisma** — 类型安全 ORM / 需要数据库服务
**结论：** PostgreSQL + Prisma。类型安全迁移，为未来多用户预留能力

### D005: 问诊+预览页面合并
**日期：** 2026-03-28
**背景：** 原设计 /checkin 和 /preview 是两个独立页面，跳转体验碎片化
**考虑的方案：**
1. **保持两个页面** — 职责清晰 / 跳转打断对话感
2. **合并为 /session** — 流畅 / 页面逻辑复杂
**结论：** 合并为 /session。问诊→预览→确认是一个连续对话流，不应被页面跳转打断

### D006: 品牌配色
**日期：** 2026-03-27
**背景：** 需要与竞品形成视觉差异化
**考虑的方案：**
1. **暗蓝 + 星空** — Calm 风格 / 太暗，不适合白天练习
2. **橙色 + 插画** — Headspace 风格 / 不符合 Iyengar 严谨传统
3. **鼠尾草绿 + 衬线字体** — 自然大地色调 + 经典传承感 / 需要更多中性色支撑
**结论：** 鼠尾草绿 #5B8A72 + Noto Serif SC。传达经典传承感，与全部竞品差异化

### D007: CSS Token 体系
**日期：** 2026-04-01
**背景：** 瑜伽 App 和知识库管理后台需要共享品牌规范
**考虑的方案：**
1. **各自维护 Tailwind config** — 独立 / 容易分叉
2. **共享 npm 包** — 单一源 / 维护成本高
3. **CSS custom properties + 文档对齐** — 轻量 / 需要人工保持同步
**结论：** CSS custom properties。通过 DESIGN.md 作为两端的唯一真相源，Tailwind 映射到 var(--color-*)
