# 需求规范

## 产品定位
**一句话：** AI 驱动的艾扬格瑜伽语音伴侣，通过全双工对话引导个性化练习
**目标用户：** 有基础的瑜伽练习者，希望在家获得专业引导
**核心场景：** 用户打开 App → 语音问诊 → AI 编排序列 → 语音引导练习 → 记录完成

## 功能需求

### M-001: 问诊对话 (P0)
**描述：** 通过多轮语音/文字对话评估用户当前身体状况、经验水平、练习目标
**验收标准：**
- [ ] AC-1: 对话 3-5 轮后能输出用户评估（level + conditions + contraindications）
- [ ] AC-2: 识别禁忌症并排除对应体式
- [ ] AC-3: 支持语音和文字两种输入方式

### M-002: 智能编排 (P0)
**描述：** 基于问诊评估和知识库，自动编排适合用户的体式序列
**验收标准：**
- [ ] AC-1: 编排序列包含 8-15 个体式，含保持时长
- [ ] AC-2: 过滤后至少保留 3 个体式（安全红线）
- [ ] AC-3: 序列遵循"热身 → 站立 → 坐姿 → 扭转 → 恢复 → 挺尸式"结构

### M-003: 练习引导 (P0)
**描述：** AI Agent 通过语音实时引导用户完成体式练习
**验收标准：**
- [ ] AC-1: 按序列逐一播报体式名称和进入指令
- [ ] AC-2: 保持时长结束时有声音/视觉提示
- [ ] AC-3: 用户可语音说"跳过"切换下一体式
- [ ] AC-4: 全流程不超过 200 轮对话（maxTurns 安全阀）
- [ ] AC-5: 最后一个体式后引导进入挺尸式

### M-004: 记忆系统 (P1)
**描述：** 跨 session 记住用户的练习历史、偏好、身体状况变化
**验收标准：**
- [ ] AC-1: 记录每次练习的完成情况
- [ ] AC-2: 下次练习时 Agent 能引用上次记录

### M-005: 首页 (P0)
**描述：** 问候 + 快捷入口，类 claude.ai greeting 风格
**验收标准：**
- [ ] AC-1: 显示个性化问候（根据时间段）
- [ ] AC-2: 一键开始新练习

### M-006: 完成页 (P1)
**描述：** 练习结束后的总结和反馈
**验收标准：**
- [ ] AC-1: 显示本次练习统计（时长、体式数）
- [ ] AC-2: 提供简短的练习建议

### M-007: 体式图鉴 (P2)
**描述：** 可浏览的体式库，含 3D 模型和详细说明
**验收标准：**
- [ ] AC-1: 按分类浏览体式
- [ ] AC-2: 每个体式有名称、功效、注意事项

## 非功能需求

### 安全
- 禁忌症体式必须被过滤，不可出现在编排中
- Agent 不提供医疗建议，遇到疼痛相关问题引导就医

### 性能
- 语音响应延迟 < 2s（含 ASR + LLM + TTS）
- 首页加载 < 2s on 4G

### 可用性
- 练习中体式名 1-2m 距离可读
- 支持语音全程操控（无需触碰手机）

## 外部依赖
| 服务 | 用途 | 接口方式 | 可替换 |
|------|------|---------|--------|
| 火山引擎 RTC | 语音通道 | SDK + Webhook | 困难 |
| Claude API | 练习 Agent | Agent SDK | 可换其他 LLM |
| Azure AI | 问诊 LLM | REST API | 可换 |
| 知识库后端 | 体式/编排数据 | REST :8000 | 自建 |
| PostgreSQL | 持久化存储 | Prisma ORM | 可换其他 SQL |

## 暂不实现（Out of Scope）
- 多用户同时练习 — MVP 仅单用户
- 视频动作识别 — 摄像头隐私风险
- 用户自定义编排 — 用户不具备编排能力
- 社交功能 — 不是社区产品
- 离线模式 — 依赖 LLM 和 RTC
