## 编号规则

- **D001** 起始，递增分配
- **永不复用**已分配的编号（即使决策被推翻）
- 被推翻的决策标注 `[已废弃]`，并在新决策中引用旧编号
- architecture.md 的技术决策速查表中的编号必须与此文件一致

## 写作指南

**什么值得记录：**
- 技术选型（框架、数据库、第三方服务）
- 架构决策（分层方式、状态管理方案）
- 产品决策（功能取舍、优先级变化）
- 设计决策（配色方案、字体选择）— 可以放在 DESIGN.md 的 Decisions Log 中

**做：**
- 每条决策必须包含被拒绝的方案
- 用具体事实说明理由（"延迟 < 100ms" 而非 "性能更好"）
- 使用绝对日期（不写"今天"或"上周"）

**不做：**
- 不记录显而易见的选择（"我们决定用 Git 做版本控制"）
- 不记录临时性决策（"先用 console.log 调试"）
- 不在结论中留模糊空间（"可能会选 A"→ 要么选了，要么没选）

## 真实案例片段

```markdown
### D001: 语音通道选型
**日期：** 2026-03-25
**背景：** 需要实现全双工语音对话（用户说话时 AI 能听，AI 说话时用户能打断）
**考虑的方案：**
1. **WebSocket + 自建 ASR/TTS** — 灵活度高 / 工程量大，需要自己处理 VAD、回声消除
2. **OpenAI Realtime API** — 一站式方案 / 国内延迟高，且绑定 OpenAI 模型
3. **火山引擎 RTC (LLM 模式)** — 国内基础设施成熟、原生支持 ASR+TTS+VAD+打断 / SDK 文档质量一般
**结论：** 选择火山引擎 RTC，因为国内延迟最低（< 200ms RTT），原生支持全双工打断，且有 LLM webhook 模式可对接任意大模型
**风险：** SDK 文档质量一般，遇到问题时排查成本高。缓解：先跑通 demo，记录踩坑经验

---

### D006: 品牌配色
**日期：** 2026-03-27
**背景：** 需要确定应用的视觉风格，与竞品形成差异化
**考虑的方案：**
1. **暗蓝 + 星空** — Calm 风格，冥想感强 / 太暗，不适合白天练习
2. **橙色 + 插画** — Headspace 风格，亲切 / 太活泼，不符合 Iyengar 的严谨传统
3. **鼠尾草绿 + 衬线字体** — 自然大地色调 + 经典排版 / 需要更多中性色支撑
**结论：** 选择方案 3，鼠尾草绿（#5B8A72）+ Noto Serif SC。传达瑜伽经典传承感，与全部竞品的现代风形成差异
```
